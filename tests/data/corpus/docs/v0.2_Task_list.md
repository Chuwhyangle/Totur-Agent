## 1. 背景与问题

Tutor Agent 是一个以「AI 学习辅导」为主题的全栈练习项目，长期愿景是成为**面向任意领域学习者的通用学习 Agent**：支持多种导师人设、工具生态、联网搜索与 RAG 知识库。

当前版本（v0.1.x）已经跑通了对话主链路，但存在两个卡住愿景的结构性问题：

1. **工具调用是单回合的**：模型只有一次使用工具的机会（请求工具 → 执行 → 回填结果 → 强制产出最终回答），无法基于工具结果决定「再查一次」或「换个工具」。多步推理-行动（ReAct）能力缺失，后续一切工具生态（搜索、RAG 检索）都会被这个瓶颈卡住。
2. **导师人设硬编码**：系统提示词写死在 `prompt_builder` 中，只有「后端学习导师」一种角色，无法支撑「任意领域学习者」的定位。

v0.2 聚焦解决这两个问题，**不做新领域功能**。

## 2. 现状概览（v0.1.x As-Is）

### 已具备能力

| 模块 | 现状 |
|---|---|
| 后端接口 | FastAPI：`/health`、`/chat`、`/conversations/{user_id}`、`/sessions`、`/interview-jds` |
| 结构化回复 | `answer` / `next_task` / `exercise` / `checkpoints` 四段式导师回复 |
| 记忆系统 | 会话隔离（`chat_sessions`）+ 最近 6 条原始历史 + 滚动摘要（8 条旧消息触发，`SummaryService`） |
| 工具调用 | `ToolRegistry` / `ToolExecutor`（结构化错误处理）+ 1 个工具 `interview_jd_search`；**单回合** |
| 前端 | React + Vite 聊天工作台：会话侧栏、调试面板、JD 面板 |
| 持久化 | SQLite（对话、会话、摘要、面试 JD） |
| 测试 | pytest 7 个测试文件，覆盖 agent 模块、工具与 API |

### 已知局限（本版处理范围见第 3 节）

- 工具循环单回合，无多步推理行动能力 ← **v0.2 解决**
- 人设硬编码，无法泛化到其他领域 ← **v0.2 解决**
- 无真实用户体系（`user_id` 字符串区分）、无 RAG / 长期记忆、无流式输出 ← 本版不处理

## 3. 目标与非目标

### 目标

- **G1**：ReAct 多轮工具循环成为 Agent 默认执行引擎，模型可以自主决定连续多次调用工具。
- **G2**：人设系统上线——新增一个人设只需加一份配置，不改核心代码。
- **G3**：向后兼容——现有 `/chat` 请求/响应契约与前端在零改动下继续可用（`tool_trace` 允许扩展字段）。

### 非目标（明确不做，防止范围蔓延）

- 联网搜索工具（v0.3 候选）
- RAG / 长期记忆 / 语义检索（记忆层级 4-6，v0.3+）
- 真实用户注册登录体系
- 流式输出（SSE）
- 前端大改版（仅允许调试面板、人设下拉框与人设徽章等小幅适配）
- 按人设定制响应 schema——v0.2 所有人设共用现有四段式结构化回复

## 4. 用户故事

- **US1**（学习者）：当我提出需要查资料的问题时，希望 Agent 能自主多次调用工具直到信息足够，以便一次得到完整回答，而不是半截答案。
- **US2**（学习者）：我希望选择不同领域的导师人设（如后端导师、算法教练、模拟面试官），以便在不同学习场景获得合适的辅导风格。
- **US3**（开发者，即我自己）：我希望在调试面板看到每一轮「思考 → 行动 → 观察」的轨迹，以便理解和调试 Agent 行为。
- **US4**（开发者）：我希望新增一个人设只需要加一份配置，以便快速试验不同 Agent 角色。

## 5. 产品设计说明：人设

### 5.1 为什么需要人设（产品价值）

1. **不同学习场景的价值契约不同。** 学概念要"讲解 + 练习"；刷算法要"给提示、不给答案"；模拟面试要反向提问与点评。三种交互模式互相冲突，单一通用导师在每个场景都平庸。人设把一个通用品拆成多个专用品。
2. **人设承载产品差异化。** 本产品与通用聊天工具的差异在于把学习方法论编码为结构化辅导（四段式回复）。人设让产品能承载**多套**方法论，是"通用学习 Agent"定位成立的前提。
3. **人设是低成本产品实验仪器。** 每个人设对应一个产品假设（"有人需要算法教练吗？"），新增成本仅为一份配置。通过观察各人设使用量验证方向，指导 v0.3 的投入重点。

**必要性边界**：人设的必要性完全来自"通用化"愿景；若产品长期只服务单一场景（后端学习），则人设属于过度设计。

### 5.2 交互设计原则

- **默认零选择**：`tutor` 为缺省人设，不选择即可用，老用户零感知（选择本身是摩擦）。
- **切换低成本**：一个下拉框完成切换，不打断输入流。
- **状态可见**：当前人设在消息区与会话列表中始终可见，避免"我在和谁对话"的困惑。

### 5.3 前端布局

v0.2（工作台阶段）改动收敛在三处：

```text
┌────────────┬──────────────────────────────────┐
│ 会话侧栏    │  [导师人设 ▾]        ← 1 顶栏下拉框 │
│            │──────────────────────────────────│
│ ○ 算法 #3  │   ● 算法教练         ← 2 当前人设徽章│
│ ● 后端 #2  │                                  │
│ ○ 面试 #1  │   …聊天消息流…                    │
│  ↑ 3 会话   │                                  │
│  人设小标签 │──────────────────────────────────│
│ [+ 新会话]  │   [输入框________________] [发送] │
└────────────┴──────────────────────────────────┘
```

1. 顶栏人设下拉框：默认 `tutor`，选项来自 `GET /personas`
2. 消息区顶部当前人设徽章
3. 会话侧栏每条会话显示人设小标签

v0.3+（产品化阶段，本版不做）：新建会话时展示人设选择卡片（名称 + 一句话简介 + 示例问题）；会话中途**不可**切换人设——中途切换会造成上下文错乱，属于明确禁止的反模式，也是 FR2.5（会话绑定人设）的产品动机。v0.2 工作台阶段允许中途切换，用于开发调试对比。

## 6. 功能需求

### F1 ReAct 多轮工具循环 — P0

| 编号 | 需求 | 优先级 |
|---|---|---|
| FR1.1 | 循环引擎：模型输出 `tool_calls` 时执行工具并将结果追加回上下文，允许模型继续请求工具，直到产出最终回答或达到轮次上限 | P0 |
| FR1.2 | 轮次上限 `MAX_TOOL_ROUNDS` 可配置（默认 3），与记忆配置一样集中管理（参照 `memory_settings.py` 风格） | P0 |
| FR1.3 | 达到上限时，发起一次不带 tools 的收尾调用，保证用户总能得到最终回答 | P0 |
| FR1.4 | 工具失败（`ok: false`）不中断循环：错误作为观察结果回传模型，由模型决定重试、换路或直接回答 | P0 |
| FR1.5 | `tool_trace` 扩展为按轮记录（round、工具名、参数、结果摘要），`/chat` 响应保持向后兼容 | P0 |
| FR1.6 | 单条工具观察结果超长时截断（上限进配置），防止循环中 token 膨胀 | P1 |

### F2 人设系统（Prompt Profiles）— P1

| 编号 | 需求 | 优先级 |
|---|---|---|
| FR2.1 | 人设定义：`persona_id`、名称、系统提示词、简介；首批内置 3 个：`tutor`（默认，行为与现状一致）、`algorithm_coach`、`interviewer` | P1 |
| FR2.2 | `/chat` 请求新增可选字段 `persona_id`，缺省为 `tutor`，保证旧请求兼容 | P1 |
| FR2.3 | 无效 `persona_id` 返回 422，错误信息中列出可用人设 | P1 |
| FR2.4 | 新增 `GET /personas`，返回可用人设列表（id、名称、简介） | P1 |
| FR2.5 | 会话与人设绑定：会话创建时记录 `persona_id`，同一会话后续轮次默认沿用（涉及 `chat_sessions` 表迁移） | P2（可延期） |
| FR2.6 | 所有人设共用现有结构化回复 schema，不因人设改变响应格式 | P1 |
| FR2.7 | v0.2 所有人设共享全部已注册工具，不做按人设的工具白名单（`allowed_tools` 为 v0.3 候选，待工具增多后评估） | P1 |

## 7. 边界情况与异常流

| 场景 | 期望行为 |
|---|---|
| 模型在最后一轮仍请求工具 | 触发 FR1.3 收尾调用，返回最终回答 |
| 工具连续每轮都失败 | 错误逐轮回传；达到上限后收尾调用，模型基于已知信息兜底回答；trace 完整记录失败 |
| 模型请求未注册的工具名 | 沿用现有 `tool_not_found` 结构化错误，作为观察回传（不抛异常） |
| 工具参数非法 JSON | 沿用现有 `invalid_arguments` 结构化错误回传 |
| 无效 `persona_id` | 422 + 可用人设列表 |
| LLM 调用本身异常 | 与现状一致的错误处理路径，不引入新行为 |

## 8. 验收标准（可测试）

验收方式：新增 `tests/test_agent_react_loop.py`、`tests/test_personas.py`；全量回归 `pytest -q` 通过。

- **AC1**（FR1.1）：Given mock 模型第 1 轮请求工具、第 2 轮基于结果再次请求工具（不同参数）、第 3 轮给出最终回答，When 调用 `/chat`，Then 响应成功，`tool_trace` 含 2 轮记录，`answer` 来自第 3 轮输出。
- **AC2**（FR1.2/1.3）：Given mock 模型每轮都请求工具，When `MAX_TOOL_ROUNDS=3`，Then 恰好执行 3 轮工具调用后发起一次不带 tools 的收尾调用，响应仍含有效 `answer`。
- **AC3**（FR1.4）：Given 工具第 1 轮返回 `ok: false`，When mock 模型第 2 轮换参数重试成功，Then 循环继续且最终回答成功。
- **AC4**（G3 向后兼容）：现有全部 pytest 测试**不修改断言**即通过。
- **AC5**（FR2.1/2.2）：Given `persona_id=algorithm_coach`，When 调用 `/chat`，Then 发给模型的 system prompt 使用该人设内容（通过 mock 捕获 messages 断言）。
- **AC6**（FR2.3）：Given 无效 `persona_id`，Then 返回 422 且错误信息含可用人设列表。
- **AC7**（FR2.4）：`GET /personas` 返回含 `tutor` 在内的全部内置人设。

## 10. 风险与开放问题

- **R1** 多轮循环放大 token 消耗与响应延迟 → 缓解：`MAX_TOOL_ROUNDS` 默认 3 + 观察结果截断（FR1.6）。
- **R2** 弱模型可能重复请求相同工具+相同参数陷入原地打转 → 缓解：检测到重复调用时短路进入收尾调用（实现细节留给设计文档）。
- **OQ1** 人设是否会让 `next_task` / `exercise` 字段语义变味（如面试官人设下"练习题"含义变化）→ v0.2 保持 schema 不变，收集实际效果后在 v0.3 决定 schema 是否按人设演化。
- **OQ2** FR2.5 会话-人设绑定涉及表迁移，v0.2 明确顺延至 v0.3；因此会话侧栏 persona 标签本版不做持久化展示，只保留顶部当前人设选择与聊天区当前人设徽章。

- **FR2.5 会话与人设绑定**：需要 `chat_sessions.persona_id`、历史数据迁移策略、会话切换时的人设恢复规则，因此不放入 v0.2。
