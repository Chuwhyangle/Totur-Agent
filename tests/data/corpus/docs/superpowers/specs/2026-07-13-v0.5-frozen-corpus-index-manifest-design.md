# v0.5 FR5.6 + FR5.3 冻结语料与索引 Manifest 设计

- 日期：2026-07-13
- 状态：已批准，待实施
- 范围：FR5.6 评测语料冻结、FR5.3 索引 Manifest、冻结基线重标
- 不在本轮范围：FR5.1 增量索引、FR5.2 PDF、FR5.4 Milvus、FR5.7 语料分层、FR4.6 生成层 A/B 收尾

## 1. 背景与目标

v0.4 已得到 `recall@3=0.9167`、`MRR=0.8264`、`negative_accuracy=0.9091` 的检索基线，但当前评测直接依赖持续变化的 `docs/**/*.md` 和本地已有 Chroma 索引，因此 fresh clone 后无法证明该数字对应哪一份语料、哪组分块参数和哪个 Embedding 模型。

本轮按既定强约束顺序完成：

```text
FR5.6 评测语料冻结
  -> FR5.3 索引 Manifest
  -> 在冻结语料上重跑并记录基线
```

完成后必须能回答：

1. 本次指标使用了哪些源文件？
2. 每个文件的内容 hash 和 chunk 数是多少？
3. 使用了哪个 Embedding 模型及哪些分块参数？
4. 相同语料和参数是否得到相同 Manifest 指纹？
5. 评测是否完全脱离活跃 `docs/` 和正式 `chroma_db/`？

## 2. 方案选择

采用“共享索引构建服务”方案：正式索引与冻结评测共用同一套语料发现、分块、Embedding、Chroma rebuild 和 Manifest 生成逻辑。

不采用：

- 在两个脚本中复制构建逻辑：容易发生参数和路径漂移。
- 提交预计算向量或完整 Chroma 数据：仓库体积大且不能验证 fresh clone 重建链路。

## 3. FR5.6：完整冻结语料

把设计批准时当前所有 `docs/**/*.md` 完整复制到：

```text
tests/data/corpus/docs/
```

保留原始相对路径，包括 `docs/superpowers/**`、`docs/tools/**` 等干扰文档。冻结快照不是精选知识集，目的是保留 v0.4 当时真实的目标文档与干扰项环境。

构建冻结索引时，以 `tests/data/corpus/` 为语料根，因此写入 Chroma 的逻辑来源仍为：

```text
docs/api-design.md
docs/tools/interview-jd-search-tool-spec.md
```

现有 `tests/data/retrieval_eval.jsonl` 无需改写 expected sources。

### 3.1 快照更新规则

- 活跃 `docs/` 变化不会自动同步到快照。
- 只有显式版本评审才允许更新快照。
- 快照更新必须同时重跑基线、记录新 Manifest 指纹和决策原因。
- 评测默认使用冻结快照，不允许静默回退到活跃 `docs/`。

## 4. 共享索引构建服务

新增：

```text
app/services/knowledge_index_builder.py
```

职责：

1. 从给定语料根递归发现 Markdown 文件；
2. 按 POSIX 相对路径稳定排序；
3. 读取 UTF-8 原始字节和文本；
4. 计算逐文件 SHA-256；
5. 调用现有 `chunk_markdown()`；
6. 统计逐文件及总 chunk 数；
7. 通过 `EmbeddingClient` 按现有批大小生成向量；
8. 调用注入的 `KnowledgeRepository.rebuild()`；
9. 返回构建结果及 Manifest。

正式建库：

```text
docs/
  -> shared builder
  -> Chroma PersistentClient
  -> chroma_db/learning_notes
  -> chroma_db/index_manifest.json
```

冻结评测：

```text
tests/data/corpus/
  -> shared builder
  -> Chroma EphemeralClient
  -> 临时 learning_notes collection
  -> 内存 Manifest
  -> retrieval evaluation
```

冻结评测不得写入或读取正式 `chroma_db/`。

## 5. FR5.3：Manifest 模型

新增：

```text
app/services/index_manifest.py
```

Manifest schema version 1：

```json
{
  "schema_version": 1,
  "fingerprint": "sha256:...",
  "collection_name": "learning_notes",
  "built_at": "2026-07-13T08:00:00+00:00",
  "embedding": {
    "model": "text-embedding-v4",
    "dimensions": 1024
  },
  "chunking": {
    "chunk_size": 512,
    "chunk_overlap": 50
  },
  "corpus": {
    "root": "tests/data/corpus",
    "file_count": 24,
    "chunk_count": 590,
    "files": [
      {
        "path": "docs/api-design.md",
        "content_sha256": "...",
        "chunk_count": 18
      }
    ]
  }
}
```

约束：

- 不记录 API key。
- 文件路径统一使用 `/`。
- 文件记录按路径排序。
- `dimensions` 从实际 Embedding 结果获取；空语料不允许构建。
- Manifest 中的 chunk 总数必须等于实际写入 Chroma 的数量。

## 6. 稳定指纹

指纹包含：

- schema version；
- collection name；
- Embedding model；
- Embedding dimensions；
- chunk size 和 overlap；
- 排序后的文件路径；
- 每个文件的内容 SHA-256；
- 每个文件的 chunk 数；
- 文件总数和 chunk 总数。

指纹不包含：

- `built_at`：每次构建都会变化；
- `corpus.root`：不同机器的工作目录可能不同。

计算方式：

```text
稳定字段
  -> canonical JSON（sort_keys=True、固定 separators、UTF-8）
  -> SHA-256
  -> "sha256:<hex>"
```

相同语料、模型和参数重复构建必须产生相同指纹；任何被纳入稳定字段的内容变化都必须改变指纹。

## 7. Manifest 持久化与失败语义

正式建库仅在下列步骤全部成功后写 Manifest：

```text
语料读取
  -> 分块
  -> Embedding
  -> Chroma rebuild
  -> Manifest 校验
  -> 写临时 JSON
  -> 原子替换 chroma_db/index_manifest.json
```

Embedding、Chroma 或 Manifest 校验失败时：

- CLI 返回非零退出码；
- 不写入新 Manifest；
- 不允许新 Manifest 描述一个未成功构建的索引。

当前索引模式始终全量重建，所以 Embedding 模型或 chunk 参数变化天然触发全量重建；增量模式一致性留给条件项 FR5.1。

## 8. Manifest 查询命令

新增：

```text
scripts/show_index_manifest.py
```

命令：

```powershell
.\.venv\Scripts\python.exe scripts\show_index_manifest.py
.\.venv\Scripts\python.exe scripts\show_index_manifest.py --files
.\.venv\Scripts\python.exe scripts\show_index_manifest.py --json
```

默认读取 `chroma_db/index_manifest.json`。摘要输出 fingerprint、collection、Embedding 模型和维度、chunk 参数、文件数、chunk 数及构建时间。`--files` 输出逐文件 path/hash/chunk count，`--json` 输出完整 JSON。

文件不存在、JSON 无效或 schema 校验失败时，命令返回非零退出码和明确错误。

## 9. 冻结评测流程

修改 `scripts/run_retrieval_eval.py`，默认行为改为：

```text
读取 tests/data/corpus/
  -> 加载真实 Embedding 配置
  -> shared builder 构建 EphemeralClient 临时索引
  -> 执行现有 35 条 retrieval_eval.jsonl
  -> 输出指标、语料统计和 Manifest 指纹
```

人类可读输出及 `--json` 均增加：

- corpus 标识；
- Manifest fingerprint；
- corpus file count；
- corpus chunk count；
- Embedding model。

保留显式 `--use-existing-index` 调试参数。只有传入它时才评测正式本地索引；默认基线评测必须走冻结语料。

## 10. 测试策略

严格按 TDD：先写失败测试、确认因缺少功能而失败，再写最小实现。

新增：

```text
tests/test_index_manifest.py
tests/test_knowledge_index_builder.py
tests/test_frozen_corpus_eval.py
```

### 10.1 Manifest 测试

- 原始字节 SHA-256 正确；
- 文件发现顺序不影响指纹；
- 构建时间和 corpus root 不影响指纹；
- 文件内容、chunk 参数、模型或维度变化会改变指纹；
- JSON 读写一致；
- 无效 schema 有明确错误；
- 原子写入只在成功后替换目标文件。

### 10.2 Builder 测试

- 递归读取 Markdown、忽略非 Markdown；
- 相对路径稳定排序并保留 `docs/...` 来源；
- 逐文件和总 chunk 统计正确；
- Embedding 按配置批处理；
- 空语料明确失败；
- Embedding 数量或维度不一致明确失败；
- 返回 Manifest 与实际入库数量一致。

### 10.3 冻结评测测试

- `tests/data/corpus/docs/` 存在并包含批准时的完整 Markdown 集合；
- 评测使用 EphemeralClient；
- 不读取活跃 `docs/`；
- 不写本地 `chroma_db/`；
- 输出包含 Manifest 指纹；
- 相同快照重复构建得到相同指纹。

单元测试使用确定性假 Embedding，不访问网络。最终重标基线使用 `.env` 中真实 Embedding 配置。
