# v0.4 RAG 实现进度

这份文档记录 v0.4 RAG「查得准」阶段的清单、实现进度和评测结果。主规划见 `docs/v0.3-0.5 RAG.md`。

## v0.4 任务清单

- [x] FR4.1：建立检索评测集 `tests/data/retrieval_eval.jsonl`，包含 35 条用例，其中 24 条正例、11 条负例。
- [x] FR4.2：新增评测脚本 `scripts/run_retrieval_eval.py`，输出 `recall@k`、`MRR`、负例准确率，并做一次手算 cosine 与 Chroma top-1 对拍。

### 决策

- `SIMILARITY_THRESHOLD` 从 `0.35` 调整为 `0.45`。
- 理由：`0.45` 在保持 `recall@3=0.9167`、`MRR=0.8264` 不下降的同时，把负例准确率从 `0.0909` 提升到 `0.9091`。`0.55` 的 overall accuracy 相同，但 recall 已下降到 `0.8750`，因此不作为当前默认值。

### 当前失败用例

- `rag_pos_006`：查询“v0.3 真实建库一共生成了多少 chunks？”时没有命中 `docs/v0.3-rag-progress.md`，更倾向命中规划文档。
- `rag_pos_020`：查询“POST /chat 的后端数据流是什么？”时没有命中 `docs/api-design.md`，更倾向命中前端或记忆规划文档。
- 负例中仍有 1 条在 `0.45` 阈值上命中已有笔记，后续 Hybrid 或 query/语料调优时观察。

### 当前边界

- 评测集是第一版人工集合，覆盖面够启动基线，但仍需要随着真实提问继续扩充。
- 当前只校准了 `SIMILARITY_THRESHOLD`，还没有改 `CHUNK_SIZE`。
- Hybrid 检索已实现为评测模式，但没有成为线上默认。
- 下一步进入 FR4.6：种子检索对比实验。

### 生成层评分表

每个回答按 4 个字段人工打分，每项 0-2 分，总分 8 分：

| 字段 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| `source_faithfulness` | 与资料冲突、无依据或明显编造 | 大体相关，但部分依据不足 | 回答忠实于检索资料或明确说明未找到 |
| `answer_completeness` | 没回答核心问题 | 回答了部分核心点 | 覆盖问题主要方面，结构完整 |
| `tool_efficiency` | 应查未查，或明显多余/错误调用工具 | 工具使用有帮助但不够理想 | 工具使用时机合适，不浪费也不遗漏 |
| `citation_integrity` | 伪造出处，或需要出处时完全没有出处 | 出处不完整或不够精确 | 出处真实、格式正确；负例不伪造出处 |

运行命令：

人工初评分：

| 用例 | tool_only | seed_plus_tool | 观察 |
|---|---:|---:|---|
| `rag_pos_020` | 1/8 | 5/8 | seed 模式更接近项目事实，并触发了 `search_learning_notes`；但最终 answer 中嵌套 JSON 代码块，触发结构化解析 fallback，且来源偏向 seed context 的 `docs/memory-and-multi-turn-plan.md`。 |
| `rag_pos_010` | 1/8 | 6/8 | seed 模式补上了本项目 Hybrid 实验语境，明显优于通用解释；但使用笔记内容后没有按要求标注来源。 |

初步判断：

- `seed_plus_tool` 在两个正例里都提高了项目相关性、完整性和工具使用合理性。
- 当前仍不能打开默认开关。主要阻塞不是检索，而是生成层稳定性：引用笔记时来源标注不稳定，且部分回答会把 JSON/Markdown 嵌进 `answer` 字段。
- 下一步应继续小批量补 1 个负例和 1 个普通项目架构正例，确认 seed 不会让负例误答；随后再决定是先优化 prompt 约束，还是继续扩大人工评分样本。
