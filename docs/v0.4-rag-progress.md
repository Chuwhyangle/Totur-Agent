# v0.4 RAG 实现进度

这份文档记录 v0.4 RAG「查得准」阶段的清单、实现进度和评测结果。主规划见 `docs/v0.3-0.5 RAG.md`。

## v0.4 任务清单

- [x] FR4.1：建立检索评测集 `tests/data/retrieval_eval.jsonl`，包含 35 条用例，其中 24 条正例、11 条负例。
- [x] FR4.2：新增评测脚本 `scripts/run_retrieval_eval.py`，输出 `recall@k`、`MRR`、负例准确率，并做一次手算 cosine 与 Chroma top-1 对拍。
- [x] FR4.3：基于首轮评测校准 `SIMILARITY_THRESHOLD`，从 `0.35` 调整为 `0.45`。
- [x] FR4.4：实现 BM25 + 向量分数融合的 Hybrid 检索，并与纯向量基线对比；首轮结果不优于纯向量，因此不切默认。
- [x] FR4.5：rerank 条件评估完成；Hybrid 首轮 `recall@3=0.9167`，高于 0.8 触发线，因此暂不实现 rerank。
- [ ] FR4.6：种子检索对比实验；工程链路与 seed coverage 评测已完成，生成层答案质量对比待做。

### 评测结果

运行命令：

```powershell
.\.venv\Scripts\python.exe scripts\run_retrieval_eval.py --threshold-sweep
```

默认旧阈值 `0.35` 下：

- cases：35
- positives：24
- negatives：11
- top_k：3
- recall@3：0.9167
- MRR：0.8264
- negative_accuracy：0.0909
- overall_accuracy：0.6571
- manual_cosine_check：ok，delta=0.0

阈值扫描：

| threshold | recall@3 | MRR | negative_accuracy | overall_accuracy |
|---:|---:|---:|---:|---:|
| 0.25 | 0.9167 | 0.8264 | 0.0000 | 0.6286 |
| 0.35 | 0.9167 | 0.8264 | 0.0909 | 0.6571 |
| 0.45 | 0.9167 | 0.8264 | 0.9091 | 0.9143 |
| 0.55 | 0.8750 | 0.7847 | 1.0000 | 0.9143 |
| 0.65 | 0.7917 | 0.7222 | 1.0000 | 0.8571 |
| 0.75 | 0.3750 | 0.3264 | 1.0000 | 0.5714 |

### 决策

- `SIMILARITY_THRESHOLD` 从 `0.35` 调整为 `0.45`。
- 理由：`0.45` 在保持 `recall@3=0.9167`、`MRR=0.8264` 不下降的同时，把负例准确率从 `0.0909` 提升到 `0.9091`。`0.55` 的 overall accuracy 相同，但 recall 已下降到 `0.8750`，因此不作为当前默认值。

### 当前失败用例

- `rag_pos_006`：查询“v0.3 真实建库一共生成了多少 chunks？”时没有命中 `docs/v0.3-rag-progress.md`，更倾向命中规划文档。
- `rag_pos_020`：查询“POST /chat 的后端数据流是什么？”时没有命中 `docs/api-design.md`，更倾向命中前端或记忆规划文档。
- 负例中仍有 1 条在 `0.45` 阈值上命中已有笔记，后续 Hybrid 或 query/语料调优时观察。

### 当前边界

- 评测集是第一版人工集合，覆盖面够启动基线，但仍需要随着真实提问继续扩充。
- 当前只校准了 `SIMILARITY_THRESHOLD`，还没有改 `CHUNK_SIZE`。
- Hybrid 检索已实现为评测模式，但没有成为线上默认。
- 下一步进入 FR4.6：种子检索对比实验。

### 决策

- 保留 `hybrid_retriever` 和 `--mode hybrid` 作为实验能力。
- 不把 `search_learning_notes` 默认切到 Hybrid。
- 理由：在当前评测集上，Hybrid 没有提升 recall@3，MRR 与负例准确率都低于纯向量默认模式。
- 不引入 rerank。理由：FR4.5 的触发条件是 Hybrid 后 `recall@k < 0.8`，当前 Hybrid `recall@3=0.9167`，未触发。

### 决策

- Seed context 工程链路保留，但默认关闭。
- 不把 `/chat` 默认切到「种子 + 工具双通道」。
- 理由：当前 seed coverage 只能证明预检索上下文能稳定注入，不能证明生成答案质量优于仅工具通道。FR4.6 的“赢者保留”仍需要一组生成层人工评分或 LLM-as-judge 方案。

### 剩余任务

- 定义生成层评分表：来源忠实度、是否多余调用工具、回答完整性、是否伪造出处。
- 用同一组问题分别跑「仅工具通道」和「种子 + 工具双通道」，记录答案并评分。
- 只有当 seed 双通道在生成质量上胜出，才打开默认开关。
