# Tutor Agent v0.8：MCP 集成开发设计

> 状态：待审核
> 原则：先 Server 后 Client、默认关闭（feature flag）、失败自动降级、评测门禁通过后再默认开启。
> 基线（v0.6 时点）：`Recall@3=0.9167`、`MRR=0.8264`、负例准确率 `0.9091`；本地工具 4 个，ReAct 循环带轮次 / 失败 / 预算三重上限。

## 1. 背景

系统已具备完整的"私有工具协议"：`ToolRegistry`（OpenAI 格式 schema）+ `ToolExecutor`（结构化错误信封 `{ok, error, message}`）+ `ReactOrchestrator`（轮次上限、失败上限、观察截断、web_search 预算与证据台账）。这套边界与 MCP 的 `tools/list` + `tools/call` 语义一一对应，是低成本对接标准协议的理想起点。

v0.8 定位：**不重写现有 agent，在其外围加两块协议层**——

1. **对外**：把检索 / JD 分析 / 测验能力包装为 MCP Server，供 Claude Code、Cursor 等任意 host 消费；
2. **对内**：为 ReAct 编排器增加 MCP Client 适配层，热接入第三方工具生态（M+N 的两端都亲手实现）。

## 2. 现状盘点（v0.8 依赖的资产）

| 资产 | 位置 | 在 MCP 中的角色 |
|---|---|---|
| 工具注册表（4 工具 + OpenAI schema） | `app/services/agent/tools/registry.py` | Server 端 tools 素材；Client 端 schema 合并目标 |
| 结构化错误信封 | `app/services/agent/tools/executor.py` | 内外工具统一的执行契约 |
| ReAct 编排（轮次/失败/预算上限） | `app/services/agent/react_orchestrator.py` | Client 适配层的宿主 |
| Hybrid 检索 + 重排 + 分片路由 | `app/services/hybrid_retriever.py`、`shard_router.py`、`reranking.py` | `search_notes` 工具底座 |
| 语料指纹 Manifest | `app/services/index_manifest.py` | `manifest://index` 资源 + 变更通知触发器 |
| 语料 / 评测报告 / 设计文档 | `corpus/`、`reports/`、`docs/` | Resources 原语素材 |
| 三个 persona 教学提示词 | `app/services/agent/personas.py`、`prompt_builder.py` | Prompts 原语素材 |

## 3. 目标

- **G1 三原语全覆盖**：tools / resources / prompts 各有 ≥2 个生产级实现，全部来自真实业务资产，非玩具示例。
- **G2 双角色**：同一项目既是 MCP Server（对外供能）又是 MCP Host/Client（对内扩展工具库）。
- **G3 双传输**：stdio（本地开发）与 Streamable HTTP（挂载于现有 FastAPI 进程）。
- **G4 动态性**：语料重建后自动发出 `resources/list_changed`；分片增减反映到 `tools/list`。
- **G5 可观测**：工具调用打点区分 internal / mcp 通道，支撑 P95 分析。
- **G6 面试可演示**：10 分钟内完成"在 Cursor 里敲 `/quiz` → 打到自家检索栈"的现场演示。

## 4. 非目标

第一版不做：

- 不把内部 ReAct 循环改为经由自家 MCP Server 调用（进程内直调保留，MCP 仅作对外契约）。
- 不实现完整 OAuth 2.1 授权服务器（静态 bearer token + 文档化演进路径）。
- 不做 MCP Registry 发布与 server 打包分发。
- 不在 CI 中调用真实外部 MCP server / 外部 API。
- 不借 v0.8 之机重构现有工具实现。
