# v0.2 复盘：Agent 核心补完

日期：2026-07-06

## 1. 本版目标

v0.2 的目标不是扩展新业务领域，而是补齐 Agent Core 的两个底层能力：

- ReAct 多轮工具循环：让模型可以基于工具观察继续行动，而不是只拥有一次工具调用机会。
- Prompt Persona 人设系统：让导师角色从硬编码 prompt 变成可配置能力，为“通用学习 Agent”留出空间。

## 2. 已完成

- 新增 `ReactOrchestrator`，把工具调用从单回合升级为可配置的多轮 ReAct loop。
- 新增 `MAX_TOOL_ROUNDS`、`MAX_TOOL_FAILURES`、`TOOL_OBSERVATION_MAX_CHARS`，把循环边界集中到配置。
- 达到工具轮次或失败上限后，使用 no-tools final call 让模型收尾，而不是返回静态 fallback。
- `tool_trace.calls[]` 增加 `round`，同时保持旧的 `used + calls` 结构。
- 新增后端人设系统，内置 `tutor`、`algorithm_coach`、`interviewer`。
- `/chat` 支持可选 `persona_id`，旧请求默认使用 `tutor`。
- 新增 `GET /personas`，前端从后端读取可用人设。
- 前端顶栏新增人设下拉框，聊天区顶部新增当前人设徽章。
- 前端 `/chat` 请求体会发送当前 `selectedPersonaId`。
- 前端工具 trace 摘要展示 ReAct 轮次。

## 3. 关键取舍

### FR2.5 延期

会话与人设绑定延期到 v0.3。原因是它不只是“多传一个字段”，而是会改变数据模型和产品语义：

- 数据库需要给 `chat_sessions` 增加 `persona_id`。
- 历史会话需要迁移策略。
- 切换会话时，前端应该恢复会话绑定的人设，而不是继续使用当前全局选择。
- 会话中途是否允许切换人设，需要明确产品规则。

因此 v0.2 只保证“当前选择影响后续 `/chat` 请求”，不声明“会话已经绑定人设”。

### 会话侧栏标签不做假实现

侧栏每条会话显示 persona 标签依赖 FR2.5 的持久化语义。v0.2 没有落库，所以不做轻量假标签，避免 UI 暗示一个并不存在的绑定关系。

## 4. 学到的东西

- ReAct 的重点不是“调用工具”，而是把“模型决策 -> 工具观察 -> 再决策”的循环边界管住。
- 工具失败应该进入观察，而不是直接中断主流程；但程序侧仍需要 `MAX_TOOL_FAILURES` 这种硬上限。
- 人设最好由后端统一拥有，前端只传 `persona_id`。这样 prompt 规则、默认值和无效参数处理都集中在一个地方。
- 向后兼容不等于不改字段；像 `tool_trace.calls[].round` 这种扩展字段可以让新 UI 获得更多信息，同时不破坏旧调用方。
- 前端状态先保持轻量足够：当前阶段只需要 `personas`、`personasStatus`、`selectedPersonaId`，不需要提前引入复杂状态管理。

## 5. 下一步

- v0.3 评估 FR2.5：给会话绑定人设，并处理会话切换时的人设恢复。
- 继续扩展工具生态前，先补一份工具注册与工具错误语义文档。
- 当工具数量增加后，再评估按人设设置 `allowed_tools`。
- 观察三个人设在四段式回复 schema 下的实际效果，再决定是否需要按人设演化响应结构。
