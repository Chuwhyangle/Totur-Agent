# Tutor Agent RAG v0.5：Reranker 开发设计

> 状态：待审核  
> 原则：先实现、后评测、默认关闭、失败自动降级。  
> 当前基线：`Recall@3=0.9167`、`MRR=0.8264`、负例准确率 `0.9091`。

## 1. 背景

当前系统已经完成 Markdown 分块、Embedding、Chroma、Vector/BM25 Hybrid 检索、阈值过滤、Manifest 和冻结评测。下一步增加 Reranker，解决“正确 chunk 已进入候选池，但没有排进最终 Top-3”的问题。

```text
Vector / Hybrid 召回 Top-10
→ Reranker 重新打分
→ 返回 Top-3
→ search_learning_notes
→ Agent 回答
```

Reranker 只能调整候选顺序，不能找回候选池之外的文档。如果正确来源没有进入 Top-10，应优化语料、分块、Embedding、query 或召回策略。

## 2. 目标

- 支持 Vector、Hybrid 两种召回后的重排。
- Provider 可替换，不绑定单一 API 或模型。
- 默认关闭，不改变现有线上行为。
- 超时、限流、配置错误或响应非法时无损退回原排序。
- 分开保存 retrieval score 与 rerank score。
- 扩充语料后通过质量、延迟和成本评测决定是否默认开启。

## 3. 非目标

第一版不做：

- 使用通用 LLM 逐条判断相关性。
- 多个 Reranker 并行投票。
- 使用未经校准的 rerank score 拒绝文档。
- 同时修改分块、Embedding、检索阈值和 Hybrid 权重。
- 在 CI 测试中调用真实外部 API。
- 默认开启重排。

## 4. 总体架构

```text
search_learning_notes(query)
        ↓
query embedding
        ↓
Vector / Hybrid Candidate Top-10
        ↓
retrieval threshold
        ↓
ENABLE_RERANKING？
   ├─ False → 原排序 → Top-3
   └─ True
        ↓
RerankingService
        ↓
RerankerClient
        ↓
Provider 批量打分
        ↓
校验 index / score
        ↓
按 rerank score 排序
        ↓
Top-3
```

职责：

- `search_learning_notes`：候选召回、调用重排、生成工具结果。
- `RerankingService`：候选构造、稳定排序、失败降级。
- `RerankerClient`：鉴权、HTTP、超时、错误映射和响应转换。
- Provider：批量计算 query-candidate 相关性。
- 评测脚本：对比无重排与重排的质量、延迟和成本。

## 6. 配置

`app/config.py`：

```python
@dataclass(frozen=True)
class RerankerConfig:
    provider: str
    api_key: str
    base_url: str
    model: str
    timeout_seconds: float
```

`app/services/rag_settings.py`：

```python
ENABLE_RERANKING = False
RERANK_CANDIDATE_K = 10
RERANK_MAX_TEXT_CHARS = 2000
RERANK_TIMEOUT_SECONDS = 5
RERANK_MAX_RETRIES = 0
```

`.env.example`：

```dotenv
RERANK_API_KEY=
RERANK_BASE_URL=
RERANK_MODEL=
RERANK_TIMEOUT_SECONDS=5
```

配置约束：

- Embedding 与 Reranker 独立配置。
- API Key 只从环境变量读取。
- Client 懒加载；关闭重排时不得读取配置或创建网络连接。
- MVP 不自动重试，防止放大 P95；后续根据真实失败率决定是否增加一次短重试。

## 7. 内部数据模型

```python
@dataclass(frozen=True)
class RerankCandidate:
    index: int
    title_path: str
    text: str
    retrieval_score: float


@dataclass(frozen=True)
class RerankScore:
    index: int
    score: float


@dataclass(frozen=True)
class RerankOutcome:
    hits: list[KnowledgeHit]
    scores_by_index: dict[int, float]
    applied: bool
    latency_ms: int | None = None
    provider: str | None = None
    model: str | None = None
    fallback_reason: str | None = None
```

Provider 只允许返回候选 index 和 score，不能修改 chunk 的来源、标题和正文。

## 8. Client 接口

```python
class RerankerClient:
    def rerank(
        self,
        query: str,
        candidates: list[RerankCandidate],
        *,
        top_n: int,
    ) -> list[RerankScore]:
        ...
```

Client 必须：

- 一次批量发送全部候选，禁止逐条串行调用。
- 设置连接和读取超时。
- 校验 index 为整数、不重复、不缺失、不越界。
- 拒绝 NaN、Infinity 和非数字 score。
- 将 Provider 异常转换为统一 `RerankerError`。
- 错误日志不得包含 API Key、完整响应体或文档正文。

错误类型：

```text
rerank_not_configured
rerank_timeout
rerank_rate_limited
rerank_auth_failed
rerank_provider_error
rerank_invalid_response
```

## 9. 候选构造

发送给 Reranker 的文本：

```text
{title_path}\n{content}
```

规则：

- 保留标题路径，提高章节语义。
- 正文最多 2000 字符，采用确定性截断。
- 不发送完整会话历史、用户身份、绝对文件路径和无关 chunk。
- 首个模型必须支持中英文混合内容。
- Provider token 上限优先于字符上限。

## 10. 候选池和阈值

MVP 使用保守策略：

```text
召回 Top-10
→ retrieval_score >= 0.45
→ 重排
→ Top-3
```

优点：保持当前负例过滤行为。  
限制：低于 0.45 的正确候选无法被 Reranker 挽救。

后续可以实验 query-level gate：

```text
max(retrieval_score) < 0.45 → 返回空
否则对完整 Top-10 重排
```

该实验只有在扩展评测集上提升排序、且不明显降低负例准确率时才允许保留。

## 11. 排序和分数规则

成功重排时：

```text
1. rerank_score 降序
2. retrieval_score 降序
3. 原候选 index 升序
```

输出示例：

```json
{
  "title": "RAG 架构设计",
  "source": "docs/rag.md",
  "similarity": 0.73,
  "match_score": 73,
  "rerank_score": 0.91
}
```

注意：

- `similarity` 是原检索分数。
- `rerank_score` 只用于当前 query 的候选排序。
- cosine、Hybrid score 和 rerank score 不是同一量纲。
- 不把 rerank score 解释为正确率。
- 不使用 `0.45` 过滤 rerank score。
- 不经评测，不做 retrieval score 与 rerank score 的加权融合。

## 12. 降级策略

不调用 Provider：

- `ENABLE_RERANKING=False`。
- 候选为空或只有一个。
- `top_n <= 0`。

调用失败后退回原排序：

- 配置缺失。
- 超时或限流。
- 鉴权失败。
- Provider 5xx。
- JSON 或字段结构非法。
- index 或 score 校验失败。

工具摘要：

```json
{
  "rerank_applied": false,
  "rerank_fallback_reason": "rerank_timeout"
}
```

硬要求：

- Reranker 故障不能使 `search_learning_notes` 失败。
- Reranker 故障不能使 `/chat` 返回 500。
- 降级后的结果来源、分数和顺序必须与重排前一致。

## 13. `search_learning_notes` 接入

建议拆为：

```python
hits = retrieve_candidates(query, query_embedding, candidate_k)
hits = apply_retrieval_threshold(hits)
outcome = reranking_service.rerank(query, hits, top_n=RAG_TOP_K)
items = items_from_hits(outcome)
```

`summary` 增加可选字段：

```json
{
  "returned_count": 3,
  "candidate_count": 8,
  "rerank_applied": true,
  "rerank_provider": "...",
  "rerank_model": "...",
  "rerank_latency_ms": 420,
  "rerank_fallback_reason": null
}
```

这些字段只用于调试与评测，不参与事实回答。

## 14. 测试方案

### 14.1 Client

- 正常批量请求和响应解析。
- 超时、429、401/403、5xx 错误映射。
- 重复、缺失、越界 index。
- NaN、Infinity、字符串 score。
- 日志不泄露密钥和正文。
- 全部使用 Mock，不访问真实网络。

### 14.2 RerankingService

- 关闭时不加载 Client。
- 空候选和单候选不调用 API。
- 能把低排名候选提升到前面。
- 同分时保持稳定排序。
- Provider 失败时原顺序不变。
- 最终数量不超过 Top-K。
- 来源、标题和正文不被修改。
- 候选文本截断符合限制。

### 14.3 工具集成

- 关闭重排时输出与当前版本一致。
- 开启时召回 Top-10、返回 Top-3。
- 超时时工具仍返回 `ok=true` 的检索结果。
- 原有空 query、索引缺失和 Embedding 失败行为不变。
- `summary` 正确记录应用状态和降级原因。

## 16. 验收标准

### M1：工程验收

- [ ] 默认开关为关闭。
- [ ] 关闭时不加载配置、不创建 Client、不发网络请求。
- [ ] 关闭时来源、顺序和 retrieval score 与当前基线一致。
- [ ] 开启时最多重排 10 个候选，最终最多返回 3 条。
- [ ] 所有候选通过一次批量请求打分。
- [ ] 重排只改变顺序，不改变来源和正文。
- [ ] retrieval score 与 rerank score 分开保存。
- [ ] 超时、限流、鉴权失败、5xx 和非法响应全部自动降级。
- [ ] 降级后的输出与原始排序一致。
- [ ] Reranker 故障不会造成工具失败或 `/chat` 500。
- [ ] 新增测试和全量回归通过。
- [ ] 单元测试不访问真实网络。

### M2：真实 Provider 冒烟验收

使用 5～10 个中英混合查询：

- [ ] 返回全部候选的合法 index 和 score。
- [ ] 中文、英文和代码术语不出现协议错误。
- [ ] 请求未超过 Provider 的候选数量和 token 限制。
- [ ] 错误信息不泄露密钥或文档正文。
- [ ] 至少验证一次超时或故障降级。
- [ ] 记录平均延迟、P95 延迟和单次成本。
- [ ] 冒烟完成后仍保持默认关闭。

### M3：默认启用决策门

语料扩充后，评测集建议至少包含 50 个正例和 15 个负例。必须同时满足：

- [ ] Recall@3 不低于无重排基线。
- [ ] MRR 或 nDCG@3 至少提升 0.03。
- [ ] 负例准确率下降不超过 0.03。
- [ ] 至少修复 1 个“已召回但排名靠后”的 bad case。
- [ ] 新增 P95 延迟不超过 1 秒，或符合确认后的延迟预算。
- [ ] 真实评测降级率不高于 1%。
- [ ] 没有来源错配、候选丢失或 index 映射错误。
- [ ] API 成本可以接受。

没有全部通过时继续保持默认关闭。

## 17. 注意事项

### 17.1 隐私

外部 Rerank API 会收到 query 和私有知识库 chunk，风险高于只发送 query 的 Web Search。

- 敏感语料优先使用本地 Reranker。
- 确认 Provider 的数据保留和模型训练政策。
- 不发送用户身份、完整聊天历史、密钥和无关内容。

### 17.2 延迟与并发

- 必须批量请求，禁止候选逐条串行调用。
- Reranker 必须可关闭、可超时、可降级。
- 分开记录 retrieval latency 和 rerank latency。
- 后续根据真实压力再增加 semaphore、singleflight、缓存或熔断。

### 17.3 实验隔离

- 重排实验期间不要同时修改分块、Embedding、阈值和 Hybrid 权重。
- 每次实验只改变一个变量。
- 报告必须绑定 Manifest fingerprint 和 Reranker 模型版本。
- 外部模型可能更新，重要评测应保存原始 before/after 排名。

### 17.4 当前 Hybrid 状态

当前代码中 `ENABLE_HYBRID_RETRIEVAL=True`，但旧记录显示 Hybrid 的 MRR 低于纯向量。接入 Reranker 前应先在冻结语料上重跑 Vector/Hybrid 基线，避免把 Hybrid 差异误判为 Reranker 效果。

## 18. 实现顺序

```text
1. 重跑并保存当前 Vector / Hybrid 基线
2. 编写 Client 和 Service 失败测试
3. 实现配置、数据模型和 Fake Client
4. 实现稳定排序与无损降级
5. 接入 search_learning_notes，默认关闭
6. 扩展 retrieval eval 的 --rerank 模式
7. 选择真实 Provider 做小规模冒烟
8. 语料扩充后执行完整 A/B
9. 通过决策门后再讨论默认开启
```

## 19. Definition of Done

- [ ] Client 与业务逻辑解耦，可替换 Provider。
- [ ] 默认关闭，不改变当前检索输出。
- [ ] Top-10 可批量重排并返回 Top-3。
- [ ] 两类分数分别保存。
- [ ] 所有外部错误无损降级。
- [ ] 关闭、成功和失败路径均有测试。
- [ ] 评测支持 Vector/Hybrid 与 Rerank 组合。
- [ ] 报告包含延迟、模型和 Manifest 信息。
- [ ] 隐私边界已经明确记录。
- [ ] 语料扩充后完成正式启用评审。
