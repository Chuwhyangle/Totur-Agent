## 1. 背景与问题

v0.2 已交付 ReAct 多轮工具循环（`react_orchestrator`）、人设系统（3 个内置人设）与两个工具（`interview_jd_search`、`score_jd_skill_fit`），全仓 93 个测试。Agent 的"行动能力"已就位，但**知识只有两个来源：模型参数记忆 + 面试 JD 库**。

问题：学习者积累的笔记（`docs/` 目录十几篇 markdown）Agent 完全用不上——问"我之前记的记忆分层方案是什么"，模型只能凭空编。RAG 让 Agent 接入自有语料，这也是"通用学习 Agent"愿景的必经一步。

**核心架构判断：在本项目中，RAG 不是一个系统，只是第三个工具。** 循环、注册表、观察截断、错误回传全程复用 v0.2 基建，整个规划不改循环代码（可验证，见 AC4）。

## 2. RAG 系统全景

完整 RAG 系统共 5 层，每层都有"从简到繁"的梯子。本规划的三个版本就是在这张图上有节制地爬梯：

| 层 | 初级（v0.3 采用） | 中级（v0.4/v0.5 按需） | 高级（本规划不做） |
|---|---|---|---|
| 语料层 | 本地 markdown | PDF 解析（v0.5，条件触发） | 用户上传 + 语料管理平台 |
| 索引层 | 标题分块 + 全量重建 | 增量同步、embedding 签名（v0.5，条件触发） | 语义分块、索引版本管理 |
| 存储层 | Chroma 嵌入式向量库（本地持久化） | Milvus 服务版（v0.5，学习/规模触发） | 分布式向量库 + 缓存 |
| 检索层 | 向量 top-k + 阈值 | Hybrid（BM25+向量）、rerank（v0.4） | query 改写、多路召回（DeepTutor SmartRetriever 型） |
| 质量层 | 手动验证 | **评测集：recall@k + 答案抽查（v0.4 核心）** | 线上监控、bad case 回流 |

**贯穿一切的是质量层**：工业界 RAG 的分水岭不是用了什么检索技术，而是能否度量。没有评测集，一切优化都是随机游走。因此评测集在本规划中从"以后再说"提升为 v0.4 的第一步。

## 3. 版本路线图：按"要回答的问题"切分

切分原则：不按技术层切，按问题切。每个版本回答一个问题，答完才有资格进入下一版。

| 版本 | 回答的问题 | 主题 | 预估 |
|---|---|---|---|
| v0.3 | 端到端能通吗？ | 能用：markdown → 分块 → Chroma 向量库 → 检索工具 → 出处标注 | ≈3 天 |
| v0.4 | 查得准吗？怎么证明？ | 查得准：先建评测集，评测驱动做阈值校准、分块调优、hybrid、种子检索、（可选）rerank | ≈4-5 天 |
| v0.5 | 能长期养吗？ | 可持续：增量索引、PDF 解析、语料管理、embedding 签名、Milvus 服务版评估——每项设触发条件，不默认做 | ≈3-4 天 |

## 4. 目标与非目标

### 目标（整个规划）

- **G1**：端到端可用——本地 markdown 笔记可被 Agent 检索并用于回答。
- **G2**：可溯源——检索结果带来源文件，回答标注出处。
- **G3**：零侵入——`react_orchestrator` 零改动，验证 v0.2 工具架构"即插即用"的设计承诺。
- **G4**：评测闭环——v0.4 起检索质量可度量，一切优化项凭评测数据保留或回滚。

### 非目标（整个规划都不做）

- 用户上传语料（产品化阶段，牵动权限/配额/多租户）
- 联网搜索、Docker、MCP（另行规划）
- 语义分块、query 改写、多路召回、索引版本管理、线上监控（高级梯级，当前语料规模不需要）

### 条件触发项（写进规划但不默认做）

- Rerank：hybrid 后 recall@k 仍不达标才引入（v0.4）
- Milvus（Docker 服务版）迁移：想补服务式部署/Docker 经验，或语料、并发增长时评估（v0.5）
- PDF 解析：真实出现 PDF 语料需求才做（v0.5）

## 5. 用户故事

- **US1**（学习者）：我问到自己笔记里记过的内容时，希望 Agent 检索笔记并**注明出处文件**回答，而不是凭模型记忆编造。
- **US2**（开发者）：我新增或修改笔记后，希望跑一条命令就能重建索引。
- **US3**（开发者）：我希望检索工具的行为有测试保障，且不依赖真实 embedding API 也能跑测试。
- **US4**（开发者）：我调整检索参数或策略后，希望跑一次评测脚本就能量化对比效果，而不是靠感觉。

## 6. v0.3 详细计划：能用（≈3 天）

### 步骤 1 — 离线索引链（能建库）

| 编号 | 需求 |
|---|---|
| FR3.1 | 分块器：按 markdown 标题切分；超长段落按 `CHUNK_SIZE=512` + `CHUNK_OVERLAP=50` 兜底切分（参数对齐 DeepTutor 默认，便于对照） |
| FR3.2 | Embedding 客户端：复用现有 OpenAI-compatible 配置，模型名进环境变量（`EMBEDDING_MODEL`） |
| FR3.3 | Chroma 嵌入式向量库（`chromadb`，持久化到本地目录）：集合 `learning_notes` 按 cosine 度量配置，metadata 存 source 文件路径/标题路径/created_at；外包一层薄 repository，工具与测试不直接依赖 chroma API |
| FR3.4 | `scripts/build_knowledge_index.py`：扫描 `docs/*.md` → 分块 → embedding → 删除并重建集合（幂等可重跑） |

**步骤 1 完成标志（AC1）**：跑脚本后 Chroma 集合有数据且每块带 source 元数据；重复跑不产生重复数据。

### 步骤 2 — 在线检索工具（能查到）

| 编号 | 需求 |
|---|---|
| FR3.5 | 新工具 `search_learning_notes(query)`：query → embedding → Chroma 余弦检索 top-k（`RAG_TOP_K=3`），相似度低于 `SIMILARITY_THRESHOLD` 的结果丢弃，宁可返回"没找到"也不硬凑 |
| FR3.6 | 注册进 `ToolRegistry` 即完成接入——循环、观察截断、错误回传（`ok:false` 结构化错误）全部复用 v0.2 机制 |
| FR3.7 | 工具返回含：chunk 文本、source、相似度分；索引未构建时返回 `index_not_built` 结构化错误 |
| FR3.8 | RAG 参数集中配置（沿用 `memory_settings.py` 风格） |

**步骤 2 完成标志（AC2）**：mock embedding + 临时 Chroma 集合下，工具按相似度正确排序返回 top-3 含 source；集合缺失/空 query 返回结构化错误不抛异常。

### 步骤 3 — 提示词接入与收尾（能用对）

| 编号 | 需求 |
|---|---|
| FR3.9 | system prompt 增加检索指引：何时该调用笔记检索（用户问"我的笔记/之前记过/我们讨论过"类问题）、回答须以 `（来源：文件名）` 格式标注出处 |
| FR3.10 | 测试：分块器单测、mock embedding 的相似度排序测试、工具注册与错误路径测试（`tests/test_knowledge_search.py`） |
| FR3.11 | 端到端手动验证 + README 与 `agent-architecture.md` 更新 + 复盘 |

**步骤 3 完成标志（AC3/AC4）**：真实提问"我笔记里关于记忆分层的方案"，Agent 调用工具且回答含出处；全量回归通过；`git diff` 显示 `react_orchestrator.py` 零改动（G3 的硬验证）。

## 7. v0.3 边界情况与异常流

| 场景 | 期望行为 |
|---|---|
| 索引未构建 / 集合为空 | 工具返回 `index_not_built` 结构化错误，模型据此告知用户并兜底回答（复用 v0.2 FR1.4 机制） |
| 检索无高于阈值的结果 | 返回"未找到相关笔记"，禁止硬凑 top-k 低质结果 |
| embedding API 调用失败 | 工具返回 `embedding_failed` 结构化错误回传循环，不中断请求 |
| query 为空 | `invalid_arguments` 结构化错误（复用 executor 现有校验风格） |
| 检索块超长 | 复用循环的观察截断（v0.2 FR1.6），无需新逻辑 |

## 8. v0.3 验收标准（AC1–AC4）

- **AC1**（步骤 1）：索引脚本产出带 source 元数据的 Chroma 集合，幂等可重跑。
- **AC2**（步骤 2）：mock embedding 下相似度排序正确、top-k 与阈值生效、错误路径全部结构化返回。
- **AC3**（步骤 3）：端到端提问触发工具调用，回答含出处标注。
- **AC4**（G3）：`react_orchestrator.py` 零 diff；现有 93 个测试不修改断言全部通过。

## 9. v0.4 详细计划：查得准（≈4-5 天）

核心问题："查得准吗？怎么证明？" 原则：**先建尺子，再做优化**——每个优化项必须在评测集上证明收益才保留，否则回滚。

| 编号 | 需求 |
|---|---|
| FR4.1 | 检索评测集：30-50 个「提问 → 应命中的笔记文件/块」用例，含 10+ 个"笔记里没有、应答不出"的负例；存 `tests/data/retrieval_eval.jsonl` |
| FR4.2 | 评测脚本 `scripts/run_retrieval_eval.py`：输出检索层指标（recall@k、MRR）；生成层先用人工抽查打分建立基线；脚本内手算一次余弦相似度并与 Chroma 结果对拍（补底层原理课） |
| FR4.3 | 基线校准：用评测集校准 `SIMILARITY_THRESHOLD` 与 `CHUNK_SIZE`，回写 OQ1 |
| FR4.4 | Hybrid 检索：BM25（`rank_bm25`，语料取自 Chroma 集合文档）与向量分数加权融合，与纯向量做评测对比（DeepTutor 的默认检索模式即 hybrid） |
| FR4.5 | Rerank（条件触发）：仅当 hybrid 后 recall@k 仍低于目标线（0.8）才引入（DeepTutor 默认也无 rerank） |
| FR4.6 | 种子检索对比实验（模仿 DeepTutor 双通道）：进入循环前用用户问题预检索、注入带截断上限的种子上下文块；在评测集上对比「仅工具通道」vs「种子+工具双通道」的答案质量，赢者保留 |

**v0.4 完成标志**：recall@k 有基线数字，历次改动的指标变化有记录；每个保留下来的优化项都有评测证据。

## 10. v0.5 详细计划：可持续（≈3-4 天）

核心问题："能长期养吗？" 原则：**每项都有触发条件，不默认做**，防止简历驱动堆料。

| 编号 | 需求 | 触发条件 |
|---|---|---|
| FR5.1 | 增量索引：按文件内容 hash 检测变更，只重算变更文件 | 全量重建的耗时/费用开始影响迭代 |
| FR5.2 | PDF 解析接入 | 真实出现 PDF 语料需求 |
| FR5.3 | 语料清单命令：列出已索引文件、块数、最近更新时间 | 无条件（观测基础能力） |
| FR5.4 | Milvus（Docker standalone）迁移评估：从嵌入式 Chroma 迁到服务式向量库，repository 接口不变只换实现；顺带补 Docker 部署经验 | 想补服务式部署/Docker 经验，或语料、并发增长；满足其一即评估 |
| FR5.5 | Embedding 签名：索引记录 embedding 模型标识，检测到变更强制全量重建（DeepTutor 的 index_versioning 思路） | 更换 embedding 模型时 |

**v0.5 完成标志**：改一篇笔记后重建只处理该文件；语料清单可查询；未触发的项留有决策记录而非默默消失。

## 11. 对标参考：DeepTutor 的 RAG 怎么做

对标其**默认 llamaindex 管线**（另有 graphrag/lightrag/pageindex 可切换，属多引擎选型，不学）。

| 环节 | DeepTutor 做法 | 对应本规划 |
|---|---|---|
| 分块 | LlamaIndex `SentenceSplitter`，chunk_size=512 / overlap=50，可配置且有边界钳制 | v0.3 FR3.1（参数已对齐） |
| 存储 | FAISS 本地二进制索引；faiss 未安装时优雅退化为 JSON `SimpleVectorStore`（线性扫描） | 我们 v0.3 用 Chroma 嵌入式——同属"本地无服务"路线，但换来真实向量库 API；v0.5 评估 Milvus 服务版，覆盖"嵌入式 vs 服务式"两种工业形态 |
| 检索 | **默认 hybrid**：向量 + BM25 融合（子检索器各取 2 倍候选再融合）；BM25 依赖缺失时自动退化纯向量；默认无 rerank | v0.4 FR4.4/FR4.5，两个决策都被验证 |
| Agent 接入 | **双通道**：① 进循环前用用户问题预检索，注入 `[Knowledge Base Context]` 种子块（每 KB 有字符截断上限）② 循环中模型仍可调 `rag` 工具追查 | 我们 v0.3 仅工具通道；种子通道进 v0.4 FR4.6 对比实验 |
| 索引运维 | embedding 签名 + 索引版本（换 embedding 模型触发重建）、索引预检 preflight/probe | v0.5 FR5.5；我们的 `index_not_built` 结构化错误是 preflight 的最小版 |
| 检索评测 | 未见 recall@k / MRR 评测 harness | **v0.4 评测闭环是我们相对它的差异化** |

明确不学的部分（解决的是多租户/多引擎问题，与单人项目规模不匹配）：多管线工厂、SmartRetriever 多查询聚合（LLM 生成 3 个查询变体并行检索再 LLM 聚合，每次检索多 2-4 次 LLM 调用）、多 embedding 供应商适配器层、KB 绑定 provider 机制。

对照学习方法：每完成本规划一个环节，回到上表读 DeepTutor 同环节源码（`deeptutor/services/rag/`），最简实现与工程化版本之间的落差即为学习清单。

## 12. 风险与开放问题

- **R1** embedding API 不可用会阻塞 v0.3 → 缓解：客户端接口与实现分离，测试全程 mock，真实调用只在建索引脚本和线上检索。
- **R2** 分块质量差导致检索不准 → 缓解：CHUNK_SIZE/OVERLAP 进配置，调参不改代码；v0.4 评测不达预期时优先调分块而非加 reranker。
- **R3** 语料太少导致检索价值不明显 → 现有 `docs/` 十几篇高质量 md 足够 MVP 验证；后续每写复盘笔记语料自然增长。
- **R4** 评测集用例质量决定 v0.4 一切结论的可信度 → 缓解：用例优先从真实使用中积累（自己真的问过的问题），负例占比 20%+。
- **R5** 对照 DeepTutor 学习可能引发"抄作业式堆功能" → 缓解：只有出现在本 PRD 的 FR 才实现；DeepTutor 的其他能力记入"明确不学"清单。
- **R6** 引入 `chromadb` 依赖（API 迭代较快），且跳过了手写相似度的底层练习 → 缓解：repository 薄封装隔离 API 变动；底层原理由 FR4.2 的余弦对拍补课。
- **OQ1** `SIMILARITY_THRESHOLD` 初值定多少 → 已在 v0.4 首轮评测校准为 `0.45`。依据：35 条评测集上 `0.45` 保持 `recall@3=0.9167`、`MRR=0.8264` 不下降，同时把负例准确率从 `0.0909` 提升到 `0.9091`。
- **OQ2** 笔记中的中英混排对 embedding 检索质量的影响，v0.4 评测时观察，必要时记入改进项。
